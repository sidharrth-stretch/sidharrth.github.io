01 · Context

The Problem

"Sales and finance teams were running manual credit checks on new accounts — pulling D&B reports outside Salesforce, copying data in by hand, and making judgment calls without a consistent framework. Slow, inconsistent, and completely undocumented."

The business need was clear: bring credit enrichment inside Salesforce, automate the data fetch, and make the resulting score transparent, repeatable, and auditable. Finance teams needed to defend credit decisions to auditors — which meant the system had to produce not just a score, but an evidence trail.

Outcome Framing

The mandate went beyond "enrich fields on Account." It required a full credit intelligence workflow: automated D&B data pull → weighted composite scoring → immutable snapshot per enrichment event → on-record status visibility for the finance team.

Apex Class Responsibilities

DnbCreditEngine Entry point. Receives Account ID + scenario type, fetches the DUNS number, orchestrates the full pipeline, and persists results. Nothing else calls D&B directly.
DnbService HTTP layer. Calls the D&B Direct+ REST API using the DUNS number via Named Credential. Returns raw JSON response to the engine. No parsing or scoring logic.
DnbParser Null-safe JSON → DnbEnrichmentResult mapping. Extracts company info, financials, credit scores, and legal events. Handles country-specific schema variance with typed fallbacks on every field.
DnbCreditEvaluator Weighted composite scorer. Blends failure score (45%), delinquency score (30%), and FSI (25%) into a 0–100 composite. Assigns GREEN ≥61 / AMBER 41–60 / RED <41 with a human-readable reason.
DnbFsiScoringService Converts D&B FSI alpha codes (5A, 4A, 3A…) into a normalised 0–100 numeric score consumed by DnbCreditEvaluator. Isolated because FSI has a structurally different response schema to all L1 blocks.
Dnb_Block_Group__mdt Custom Metadata Type. One record per D&B data block with an Is_Active__c toggle. Engine reads this at runtime to build the block parameter string — no code change needed to add or remove blocks.

03 · Design Decisions

Four Architecture Decisions

Each decision below includes the reasoning, the technical mechanism, and the trade-off that was consciously accepted.

01
Metadata-driven block selection
▼

D&B's API is block-based — each data bundle is requested and billed separately. Hardcoding block names in Apex would mean a deployment every time the client's D&B contract changed or a new block was enabled.

Mechanism

Custom Metadata Type Dnb_Block_Group__mdt — one record per D&B block, with an Is_Active__c checkbox. The engine queries this at runtime and builds the block parameter string dynamically. Toggling a block on or off is a metadata record change — deployable via SFDX, no Apex deployment required.

Trade-off

Slightly more upfront complexity versus hardcoding. Paid back immediately when the client's D&B contract added the FSI L2 block mid-project — it was a new CMDT record, no code change, no release ticket.

02
Weighted composite credit score
▼

D&B returns multiple independent risk signals. No single field tells the full credit story — a company can have a high failure score but a healthy FSI rating (e.g. a young startup with strong backing). The composite had to reflect the client's actual risk priorities.

Failure Score Percentile 45%
Delinquency Score Percentile 30%
FSI Financial Strength Indicator 25%
Transparency by design

Each component score is stored as its own field on Account and on the Snapshot object. The composite is stored separately. Anyone can see exactly why an account scored 42 — which input drove it and by how much. Scores are never a black box.

Trade-off

Weights were defined with the finance team — they're embedded in DnbCreditEvaluator. If weights need changing, a developer is involved. Mitigation: expose weights as Custom Metadata in a future iteration so finance can tune them without a deployment.

03
Layered Apex with strict single responsibility
▼

A monolithic "enrich account" class would have worked for the happy path. But this integration handles international records where the D&B JSON schema varies by country — a field present in a Norwegian company response may be absent in a German one. A single class makes isolating and fixing those failures impossible without risking unrelated logic.

Defensive parsing pattern

Every field access in DnbParser goes through a null-safe guard with a typed fallback. This is not defensive for the sake of it — it's a direct response to D&B's per-country schema variance. DnbFsiScoringService is a separate class entirely because the L2 FSI block returns a structurally different JSON shape to all L1 blocks.

// Null-safe pattern in DnbParser String riskSegment = (String) safeGet(data, 'failureScore.riskSegment', ''); Decimal failurePct = (Decimal) safeGet(data, 'failureScore.nationalPercentile', 0.0); private static Object safeGet(Map<String,Object> m, String path, Object fallback) { // traverse dot-notation path with null check at each step try { return traverse(m, path); } catch (Exception e) { return fallback; } }
Trade-off

More files, more cognitive surface area. The benefit: Phase 2 parser bug fixes (six of them, all country-specific) were isolated to DnbParser and redeployed without touching the scoring engine or API service layer.

04
Finance Snapshot — immutable enrichment history
▼

Overwriting Account fields on each enrichment run loses history. The finance team needed to explain to auditors why a credit decision was made on a specific date — which requires point-in-time evidence, not current values.

Dnb_Finance_Snapshot__c

Append-only child object on Account. Each enrichment run writes one new Snapshot record containing: enrichment timestamp, all 43 D&B financial fields, each component score, and the final composite. Account stores only the latest values for UI. Snapshot stores the full history.

Why not Field History Tracking?

FHT caps at 20 fields per object, has an 18-month retention limit, and does not capture the relationship between field values at a point in time. If failure score was 72 and delinquency was 31 on a given date, FHT shows two disconnected field change records. The Snapshot captures all 43 fields as a single coherent record — exactly what an auditor needs.

Trade-off

Additional DML per enrichment run. At scale, Snapshot records accumulate fast. Handled with a configurable retention period on the object and a scheduled Database.Batchable job to archive records beyond 12 months.

04 · Observability

Logging with Nebula Logger

Rather than writing to a custom log object (which adds DML to the transaction and is wiped on rollback), this integration uses Nebula Logger — a 100% native, open-source Salesforce observability framework built on Platform Events.

Why platform events matter here

LogEntryEvent__e platform events are published asynchronously and are not rolled back with the transaction. If an enrichment run throws an exception and the DML rolls back, the log entry still publishes. A hand-rolled custom log object would be silently wiped in the same scenario — exactly when you need the evidence most.

Three logging levels in this integration

Logger.error() API service class — non-200 HTTP response. Passes the Account SObject directly, capturing full record state at failure time.
Logger.warn() — Parser Field mapping failure due to unexpected JSON structure. Tagged with country and DUNS number for filterable log queries.
Logger.warn() — Engine Scoring skipped because required block data is absent. WARN not ERROR — expected edge case for certain D&B entitlement configurations.
// Service class — HTTP failure with full SObject context Logger.error('D&B API returned non-200', accountRecord) .addTag('DnbEnrichment') .addTag(response.getStatusCode() + ''); Logger.saveLog(); // Parser — field mapping failure tagged for filtering Logger.warn('FSI field absent for country: ' + country) .addTag(dunsNumber) .addTag(country); // Async Queueable — chain logs back to parent transaction Logger.setParentLogTransactionId(originalTransactionId); Logger.info('Batch enrichment chunk complete'); Logger.saveLog();
Async transaction linking

Batch enrichment spawns multiple Queueable instances — each in its own transaction. Logger.setParentLogTransactionId() chains all child logs back to the originating Log__c record. One place to diagnose a batch run, not disconnected log fragments per Queueable instance.

05 · Production Reality

Lessons Learned

Both issues surfaced post go-live, not in sandbox.

Lesson 01

International data inconsistency — D&B schema varies by country

D&B's API documentation describes a standard JSON structure. In production, field availability varies significantly by country. Norwegian records include fields absent from German ones. Zambian records have a different risk segment taxonomy entirely. This was invisible in sandbox testing because all test data was of North American or European/Scandinavian companies with the same schema .

Fix

Defensive null-safe parsing added to every field in DnbParser with typed fallback values. A mock response factory was built for seven country profiles — including DE, ZM, GB, US — to catch country-specific failures in tests before they reach production. Phase 2 included six targeted parser bug fixes.

Lesson 02

FLS on Credit_Assessment__c fields — required manual post-deploy assignment

Permission sets covering all 43 new fields were deployed with the package. Within the first business day, users were reporting they couldn't see enriched data. Root cause: profile-level FLS on the new fields hadn't been set, and permission set assignments hadn't been fully rolled out to all relevant users before go-live.

Fix

FLS verification is now the first post-deployment check — confirmed via Setup → Object Manager → Fields → Field Accessibility against all affected profiles and permission sets before user communication goes out. Added explicitly to the deployment checklist as a non-optional step.
